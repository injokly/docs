방향성 초안 · v0.1

figma-to-react — 작업 방식 정리

Figma를 원본으로 두고 React 공통 컴포넌트/화면을 반복 가능하게 뽑아내기 위한 세팅. 아직 확정본은 아니고, 무엇을 왜 이렇게 만들어두고 있는지 공유용 문서.

1. 목적 & 기대효과

지금 이 세팅으로 줄이려는 건 크게 두 가지다. 각 목적마다 "되면 뭐가 좋은가"를 바로 붙였다.

① 기획/퍼블리싱 확인 루프 단축

Figma → 코드 초안까지를 PL 혼자 돌릴 수 있게 한다. 카피·토큰·레이아웃을 사람이 육안으로 대조하던 단계를 자동 검증으로 옮겼다.

기대효과
· 기획/디자이너 재확인 요청 감소 (카피 임의 수정·토큰 누락이 머지 전에 걸림)
· 화면 초안 생성 → 1차 리뷰까지 소요 단축
· "Figma와 다르네요" 티케팅이 Storybook 단일 URL 공유로 대체

② 사내망 핸드오프 리스크 축소

외부망에서 만든 코드가 내부망 개발팀으로 넘어갈 때, 넘기기 전에 잡을 수 있는 건 다 잡아두자는 관점. 토큰 참조·접근성·구조 규약을 정적 규칙으로 박아뒀다.

기대효과
· 내부망에서의 재작업·재해석 비용 감소
· hex/px 하드코딩, any, 상대경로 import 같은 "옮긴 뒤 발견되는" 문제 사전 차단
· 같은 컴포넌트를 두 번 만드는 중복을 인벤토리 매칭으로 방지
초안 단계라 유동적인 부분 — 지표(재작업률 등)는 아직 기준선이 없어 수치 목표 대신 방향만 썼다. 몇 주 돌려보고 고정할 예정.

2. 저장소 구조 기능별로 묶어서 본 모습

폴더를 알파벳순이 아니라 "무슨 기능인지" 기준으로 색칠해봤다.

디자인 토큰 공통 컴포넌트 화면 (Route) 자동화 스킬/에이전트 검증 스크립트 문서/스토리북
figma-to-react/ ├── design-tokens/ # Figma Variables 원본 (DTCG) │ ├── color.{light,dark}-theme.tokens.json │ ├── color-seed.value.tokens.json # 원시 팔레트, 참조 전용 │ ├── spacing / size / border-width / typo-* / effect │ └── manifest.json │ ├── scripts/ │ ├── build-tokens.mjs # tokens → global.css │ ├── verify-component.mjs # 컴포넌트 정적 규칙 │ ├── verify-screen.mjs # 화면 정적 규칙 │ ├── build-inventory.mjs # 컴포넌트 목록 DB │ └── match-figma-nodes.mjs # Figma ↔ 인벤토리 매칭 │ ├── src/ │ ├── styles/ │ │ ├── reset.css · global.css # global.css는 자동 생성 │ │ └── theme.ts # v() CSS var 헬퍼 │ │ │ ├── components/ # 현재 14종 │ │ ├── SolidButton / OutlinedButton / GhostButton / LightSolidButton / AlphaButton │ │ ├── BoxTextField / Select / Checkbox │ │ ├── Badge / Title / Typography / Header │ │ ├── BottomStickyFooter / Termbox │ │ └── _inventory.json # 자동 생성 │ │ │ └── routes/ │ └── codek-savings/terms-agree/ # 화면 예시 │ ├── .claude/ │ ├── commands/ # /component, /screen — 슬래시 커맨드 │ ├── agents/ # figma-reviewer, screen-reviewer │ └── hooks/ # verify.sh — 파일 작성 직후 자동 실행 │ ├── .storybook/ # 리뷰용 문서 사이트 ├── docs/ # component-design-summary, workflow, tokens... └── CLAUDE.md # 작업 규약 (이 세팅의 뇌)
이 그림에서 봐야 할 포인트 — 보라(토큰)는 단방향으로 흐르고, 주황(스킬/에이전트)은 외부 도구고, 빨강(검증)은 작성 직후 자동 실행된다. 이 세 덩어리가 서로 맞물리는 모습이 다음 섹션의 업무 흐름도.

3. 업무 흐름 스킬·에이전트·검증이 어디서 끼어드는지

도식 가운데 들어간 빨간 게이트가 실패하면 되돌아와서 고친 뒤에만 다음으로 간다. 이 부분이 품질의 핵심.

3-1. 공통 컴포넌트 만들기 /component [Name] [URL]

PL이 Figma URL 하나로 시작 → Storybook 리뷰까지
PL
Figma URL 전달 → (끝에서 리뷰)
스킬
/component → get_variable_defs → get_image → get_design_context
Claude
KDS 설계 합의
명명·계층·사이즈·Props 순서 → 파일 5종 생성
index / types / styles / stories / README
검증
⚡ Gate 1
정적 규칙 (hook) → 🤖 Gate 2
figma-reviewer → ✅ pass ⟲ fail → auto_fixable 즉시 수정 후 재검증
산출
src/components/{Name}/ + _inventory.json 갱신 → Storybook에서 PL 리뷰
이 흐름이 담고 있는 동작 원리 — MCP 3종(variables·image·context)을 반드시 다 본 뒤에야 파일을 쓴다. 과거에 metadata만 보고 조립해서 카피를 임의 변경한 사고(2026-04-23)가 있었고, 그 재발 방지가 이 순서의 이유다. Gate 2의 reviewer는 생성 코드가 아니라 Figma를 다시 읽어 비교한다.

3-2. 화면 만들기 /screen [path] [URL]

화면은 "이미 있는 컴포넌트 재사용"을 먼저 판정한 뒤 조립
PL
Figma URL 전달 … 매칭 리포트 승인 … 최종 확인
스킬
/screen → MCP 3종 → match-figma-nodes
분류
✅ 재사용 🆕 신규 공통 → /component 🏠 화면 전용 _Name.tsx
Claude
route.tsx 조립
loader / action / sub components
검증
⚡ Gate 1
verify-screen.mjs → 🤖 Gate 2
screen-reviewer → ✅ pass ⟲ 카피·재사용·구조 불일치 → 재수정
산출
src/routes/{path}/ → pnpm dev 실제 동작 확인
동작 원리 한 줄 요약 — 화면 흐름의 핵심은 "조립 전에 인벤토리 매칭을 먼저 한다"는 것. 같은 Button을 또 만드는 걸 막기 위해 _inventory.json과 match-figma-nodes.mjs가 중간에 끼어 있다. 토큰·검증·스킬이 이 한 흐름 안에서 각자 자리를 잡는 구조.
