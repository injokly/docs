01목적

디자이너가 Figma에서 정의한 KDS V4 (for Mobile) 컴포넌트와 화면을, 같은 의미·구조·상태를 유지한 채 React 앱으로 1:1 옮기는 워크플로우. 사람이 매번 손으로 옮길 때 발생하는 drift를 AI 슬래시 스킬과 검증 에이전트로 시스템화한다.

정성적 효과

① 디자인 일관성 보장

spec §7.9 KDS V4 first 룰로 모든 공통 컴포넌트가 KDS V4 (for Mobile) main definition에서 derive됨. AI-TEST-DS같은 product file의 instance가 아닌 canonical library를 source of truth로 강제.

② 라이트/다크 즉시 전환

token 파이프라인이 source JSON → CSS variables(:root / [data-theme="dark"]) + TS theme 객체 동시 생성. data-theme 속성 토글 한 번으로 모든 컴포넌트 색상 즉시 전환.

③ Figma 변경 추적 가능

각 컴포넌트 README에 figmaNodeId + componentKey를 frontmatter로 기재. Figma library가 버전업되면 reviewer가 drift를 자동 감지.

④ 신규 인력 onboarding 단축

컴포넌트 추가는 /figma-component <Name>, 화면 추가는 /figma-screen <URL> 한 줄. 폴더 구조·prop 순서·토큰 사용 규칙은 SKILL.md가 영구 enforce.

① 토큰 파이프라인

design-tokens/*.json
→

→

source JSON 한 곳에서 라이트/다크 모드 색상과 모드 무관 토큰을 모두 정의 → pnpm tokens:build가 CSS variables (:root, [data-theme]) + 타입 안전 TS theme 객체를 동시 생성. ESLint no-seed-token 룰이 seed 토큰 직접 참조를 차단.

② /figma-component — 컴포넌트 추가 (9-step)

  1. 인자 파싱ComponentName + 선택적 Figma URL.
  2. KDS V4 검색mcp__figma__search_design_system으로 KDS V4 (for Mobile) library에서 💠-prefixed component_set 찾기.
  3. variant property 추출get_design_context로 variant axes / boolean toggles / text props / instance swap slot을 verbatim enumerate.
  4. 토큰 매핑 + gap 처리누락 토큰을 design-tokens JSON에 추가, light/dark symmetric 유지, pnpm tokens:build 재실행.
  5. 6개 파일 생성index.tsx + types.ts + styles.ts + *.stories.tsx + *.test.tsx + README.md.
  6. Barrel + Inventorypackages/ui/src/index.ts에 export 추가, pnpm inventory:build.
  7. Lint + Test + Typecheckpnpm lint && pnpm typecheck && pnpm test 모두 0 error.
  8. figma-reviewer 검증subagent dispatch → 9개 KDS 카테고리(naming · props-order · state · touch-target · size-system · readme · resizing 등) 검사. auto_fixable issue는 자동 수정, 최대 3회 반복.
  9. 커밋feat(ui): add <Name> component (KDS V4 main <componentKey>).

③ /figma-screen — 화면 추가 (8-step)

  1. 인자 파싱 + 화면 fetchFigma URL → fileKey/nodeId 추출, design context fetch.
  2. Inventory 매칭화면이 사용하는 💠-인스턴스를 _inventory.json과 대조. 누락 발견 시 즉시 중단 + 사용자 안내 (자동 생성하지 않음 — sprawl 방지).
  3. Screen-specific 블록 식별한 화면 전용 비즈니스 블록은 _*.tsx scoped sub component로 분리. 다른 screen에서 import 금지.
  4. 화면 폴더 + 파일 생성Frame / HeaderLayout / MainLayout / *Section 구조를 Figma right-panel CSS verbose form 그대로 재현, 모든 numeric value는 theme token.
  5. App.tsx ScreenSwitcher 등록type ScreenKey 확장, toggle 버튼 추가.
  6. Lint + Test + Typecheck0 error.
  7. Dev server smokepnpm dev 부팅 확인.
  8. screen-reviewer 검증 + 커밋문자열 일치 / 컴포넌트 재사용 / 시각 구조 / 화면 폴더 규약 4단 검사 통과까지 반복 후 커밋.

05배포 — Vercel

모노레포에서 두 산출물(앱 + Storybook)을 각각 별도 Vercel 프로젝트로 배포. pnpm workspaces가 자동 감지되도록 root에서 빌드한다.

5.1 앱 (packages/app) 배포

Vercel 대시보드에서 새 프로젝트 → 이 레포 import → 다음 설정 적용.

항목값
Framework Preset Vite
Root Directory ./
Install Command pnpm install --frozen-lockfile
Build Command pnpm tokens:build && pnpm --filter @figma-react-super/app build
Output Directory packages/app/dist
Node Version 20.x

또는 레포 루트에 vercel.json을 두면 위 설정이 자동 반영: 

// vercel.json (repo root)
{
  "buildCommand": "pnpm tokens:build && pnpm --filter @figma-react-super/app build",
  "outputDirectory": "packages/app/dist",
  "installCommand": "pnpm install --frozen-lockfile",
  "framework": "vite"
}

5.2 Storybook 별도 배포

디자인 시스템 라이브 갤러리. 같은 레포로 두 번째 Vercel 프로젝트 import.

항목값
Framework Preset Other
Build Command pnpm tokens:build && pnpm --filter @figma-react-super/ui build-storybook
Output Directory packages/ui/storybook-static
Install Command pnpm install --frozen-lockfile

5.3 환경 / CI 권장

Preview deploy

모든 브랜치에 자동 preview URL. PR 본문에 link 자동 게재 → 디자이너/리뷰어가 바로 확인.

Production branch

main에 머지 시 production URL 자동 갱신. 두 프로젝트(앱·Storybook) 동일.

(선택) Chromatic

Storybook 시각 회귀 테스트. PR마다 UI diff 캡처. Vercel과 병행 가능.
